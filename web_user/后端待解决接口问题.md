# 后端待解决接口问题

更新时间：2026-06-10

本文档记录本轮用户端联调中确认不能只靠前端解决的问题。旧库 Flyway checksum 不一致的问题已按“直接重建数据库”处理，不再作为当前联调阻塞项。

## 本轮已由前端修复/规避

- 附件下载：用户端不再直接跳转到 `http://127.0.0.1:8080/api/v1/attachments/{id}/stream`，改为带登录态请求文件流并在当前页面展示具体错误。
- 评论回复：提交成功后，用户端会把新回复插入当前详情页缓存，避免后端详情接口刷新后暂时丢失子评论造成“无反应”。
- 举报反馈：举报弹窗内直接显示“原因过短 / 提交成功 / 接口错误”，不再只依赖全局提示气泡。
- 举报字段：用户端已按接口文档补发 `title`、`proofSummary`、`contactEmail`，版权证明不再拼进 `reason`，避免 `report_complaint.title` 和 `report_complaint.proof_summary` 继续为空。
- 资源库/求资源筛选：一级分类改发 `category1/cate1`，二级分类才发 `categoryId/category2/cate2`；求资源悬赏积分改发 `rewardRange`。
- 列表卡片分类：资源库和求资源 item 已改为显示 `一级分类 > 二级分类  发布者：XXX`。

## 已复测通过

- `GET /api/resources?page=1&size=10&category1=1`：只选择一级分类时可返回资源列表。
- `GET /api/requests?page=1&size=5&category1=3`：只选择一级分类时可返回求资源列表。
- `GET /api/requests?page=1&size=5&category1=3&rewardRange=0-100`：悬赏积分筛选生效。
- `POST /api/reports`：资源/评论类举报接口可提交。
- `POST /api/reports/copyright-complaints`：版权投诉接口可提交。

## 仍需后端/数据库处理

### 1. 示例附件记录对应的实际文件缺失

现象：

- `GET /api/files/{attachmentId}/download` 能返回下载元数据，例如 `downloadUrl=/api/v1/attachments/{attachmentId}/stream`。
- 继续请求附件流时，部分种子数据附件返回 `404`，错误信息为 `file does not exist`。

影响：

- 前端现在会在当前页面显示明确错误，不会再跳到 8080 页面。
- 但用户仍然无法真正下载这些示例附件，因为后端/数据库中存在附件记录，文件系统中缺少对应实体文件。

期望后端/数据处理：

- 为种子附件补齐真实文件，或把种子附件记录调整为后端实际存在的文件路径。
- 如果示例附件本来就不应下载，请后端在下载元数据接口中返回明确的不可下载状态，而不是给出一个会 404 的 stream URL。

### 2. 详情接口未返回已有子评论（2026-06-10 复核仍存在）

现象：

- `POST /api/comments` 携带 `parentId` 后能成功创建回复，数据库中也能看到 `parent_id`。
- 用户端提交成功后会把返回的新回复临时插入当前页面缓存，所以不刷新时能看到二级评论。
- 一旦刷新页面、退出重进或换浏览器，页面重新请求后端详情接口，二级评论消失。
- 代码复核确认：资源详情 `GET /api/resources/{id}` 的后端查询只返回 `parent_id IS NULL` 的一级评论，未把子评论放入 `replies`，也未以可嵌套的平铺结构返回。
- 求资源详情 `GET /api/requests/{id}` 的评论查询同样过滤 `parent_id IS NULL`，如果求资源也支持评论的二级回复，会出现同类问题。

影响：

- 前端已对“当前刚提交的回复”做乐观展示，所以提交后可立即看到。
- 但刷新页面、换浏览器、重新进入详情页后，仍依赖后端详情接口返回子评论；后端不返回时，历史回复无法恢复显示。
- 这个问题不能靠前端可靠修复，因为当前后端没有提供可查询某目标下全部评论（包含 `parentId` 非空子评论）的用户端接口。

期望后端处理：

- 资源详情接口 `GET /api/resources/{id}` 返回评论树：一级评论包含 `replies`。
- 如求资源评论也支持二级回复，则 `GET /api/requests/{id}` 同样应返回评论树或带 `parentId` 的完整评论列表。
- 或者提供独立评论列表接口，例如 `GET /api/comments?targetType=RESOURCE&targetId={id}&includeReplies=true`，返回同一目标下所有 active 评论，前端可按 `parentId` 组装树。
- 目前后端 `MappingSupport.commentMapper` 已有 `parentId` 和 `replies` 字段形态，建议在查询层补齐子评论数据，而不是让 `replies` 固定为空列表。

### 3. 个人头像缺少适合用户端使用的上传接口

现状：

- 用户资料接口可以保存 `avatar` / `avatarUrl` 这类地址字段。
- 用户端目前只能填写图片 URL。
- 通用附件上传/下载接口不适合作为头像接口：头像 `<img>` 请求无法自动携带 Authorization，且附件 stream URL 不一定是公开可访问的图片地址。

影响：

- 前端不能可靠实现“选择本地图片并保存为头像”。
- 强行把本地图片转成 base64 存入资料字段也不合适，字段长度和图片加载性能都不可控。

期望后端处理：

- 提供专用头像上传接口，例如 `POST /api/me/avatar` 或 `POST /api/v1/user/avatar`。
- 接口返回可被浏览器 `<img>` 直接访问的图片 URL，或提供前端可用的认证图片访问方案。
- 成功上传后同步更新用户资料中的头像地址。

### 4. 重复举报/版权投诉返回 500

现象：

- 同一用户对同一目标、同一举报类型重复提交时，前端收到：

```text
接口请求失败：POST /api/reports（HTTP 500，后端内部错误）：系统繁忙，请稍后再试
```

- 本地后端日志显示根因是数据库唯一索引冲突：

```text
Duplicate entry '16:RESOURCE:1:RESOURCE' for key 'report_complaint.uk_report_active_target'
Duplicate entry '16:RESOURCE:1:COPYRIGHT' for key 'report_complaint.uk_report_active_target'
```

影响：

- 用户无法知道是“已经提交过举报/投诉”，只能看到系统繁忙。
- 这不是用户端字段缺失导致；用户端已补齐 `title` 和 `proofSummary`。

期望后端处理：

- 捕获 `DuplicateKeyException`，返回明确业务错误。
- 建议 HTTP 409 或 400，message 例如：`你已提交过该举报，处理中请勿重复提交`。
- 如需要允许补充证据，应提供追加证据/更新举报的接口，而不是继续插入重复 active report。

### 5. 求资源回答缺少附件上传/绑定接口

现象：

- 原型 `user_UI/求资源详情页.html` 中，“我来回答”区域包含上传附件入口，回答列表也有带附件的展示形态。
- 当前后端 `POST /api/requests/{id}/replies` 支持 `content`、`resourceId`、`externalUrl`，可提交文字回答、关联已发布资源或外部链接。
- 当前后端文件接口 `POST /api/files/upload` 只按 `RESOURCE` 保存附件；`GET /api/files/{id}/stream` 也只按 `RESOURCE` 关联下载。
- 未发现可把附件绑定到 `REQUEST_REPLY` 的用户端接口。

影响：

- 用户端可以展示“上传附件”按钮，但不能把本地文件可靠保存到某条求资源回答下。
- 若前端强行调用现有文件上传接口，附件会成为 `RESOURCE`/临时附件，无法随回答详情返回，也无法在回答列表中下载。

期望后端处理：

- 支持求资源回答附件，例如在 `POST /api/requests/{id}/replies` 中接收 multipart 文件，或提供 `POST /api/files/upload?ownerType=REQUEST_REPLY&ownerId={replyId}`。
- `GET /api/requests/{id}` 和 `GET /api/requests/{id}/replies` 返回每条回答的附件列表。
- 附件流下载接口应支持 `REQUEST_REPLY` owner，并按权限允许相关用户下载。
